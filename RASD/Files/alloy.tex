In this Alloy analysis, we chose to focus on the Trip lifecycle. Specifically, we want to describe the evolution of a cycling session over time, from the initial recording to the final publication, ensuring that data validation steps (Goal G4) are respected.

We point out that the initial state represents the moment a trip is created but has not yet evolved through the system steps.

\subsection{Objectives}
The objective of this analysis is to verify the consistency of the \textit{Best Bike Paths} dynamic model. In particular ensuring that a trip follows the mandatory sequence of states (\textit{Recording} $\rightarrow$ \textit{Processing} $\rightarrow$ \textit{Validation}) before being published, this model also verifies that the owner of a trip cannot change over time, even if the trip entity is mutable, in the end it ensures that obstacles detected during the ride can be filtered by the user during the validation phase, but become immutable once the trip is published.


\subsection{Model Description}

The model defines the static entities (\texttt{User}, \texttt{Obstacle}) and the dynamic entity \texttt{Trip}, whose state and relations evolve over time steps.

\begin{lstlisting}[language=Alloy, caption={Signatures and Facts defined in the Alloy model}, label={lst:alloy-model}]
// Users are static entities
sig User {
    var trips: set Trip
}

// Obstacles are static entities detected during trips
sig Obstacle {}

// Enum defining the lifecycle steps of a trip
enum Step { Recording, Processing, Validation, Published }

// A Trip is linked to a single User and evolves through steps
var sig Trip {
    var owner: one User,
    var currentStep: one Step,
    var obstacles: set Obstacle
}

// FACTS

// F1: A trip always belongs to its owner's set
fact TripOwnership {
    always (all t: Trip | t in t.owner.trips)
}

// F2: System Initialization
// In the first state, all existing trips must be in Recording
fact NewTrip {
    all t: Trip | t.currentStep = Recording
}

// F2b: Creation Constraint
// Logic: If a trip exists in the next state but not now,
// it is a NEW trip and must start in Recording.
fact TripsStartInRecording {
    always (
        all t: Trip' | (t not in Trip) implies t.currentStep' = Recording
    )
}

// F3: The owner of a trip never changes
fact ImmutableOwner {
    always (all t: Trip | (t in Trip') implies t.owner' = t.owner)
}

// F4: Helper function for state transitions
fun nextState[s: Step]: one Step {
    s = Recording  => Processing else
    s = Processing => Validation else
    s = Validation => Published else
    Published  // Published stays Published
}

// F5: Describes the progression of trips
// Trips either move to the next step or stay at the current step
fact StateProgression {
    always (
        all t: Trip | t in Trip' implies (
            t.currentStep' = t.currentStep or
            t.currentStep' = nextState[t.currentStep]
        )
    )
}

// F5b: Sequential Integrity
// Explicitly forbids skipping steps to satisfy strict ordering requirements
fact NoSkippingSteps {
    always (all t: Trip | (t in Trip and t in Trip') implies (
        (t.currentStep = Recording) implies 
            (t.currentStep' = Recording or t.currentStep' = Processing)
        and
        (t.currentStep = Processing) implies 
            (t.currentStep' = Processing or t.currentStep' = Validation)
        and
        (t.currentStep = Validation) implies 
            (t.currentStep' = Validation or t.currentStep' = Published)
        and
        (t.currentStep = Published) implies t.currentStep' = Published
    ))
}

// F6: Obstacle Consistency
fact ObstacleConstraint {
    always (all t: Trip | t in Trip' implies (
        // Once published, obstacles are immutable
        (t.currentStep = Published implies t.obstacles' = t.obstacles)
        and
        // Obstacles cannot be added during Processing or Validation, only removed
        (t.currentStep in Processing + Validation implies 
            t.obstacles' in t.obstacles)
    ))
}

// F7: Liveness
fact EndTrip {
    all t: Trip | eventually (t not in Trip or t.currentStep = Published)
}
\end{lstlisting}

\subsection{Analysis and Validation}

To validate the model, we use \textbf{Predicates} to generate valid scenarios and \textbf{Assertions} to verify that no invalid states are reachable.

\subsubsection{Predicates}

\textbf{Example 1: Correct Path Scenario}

In this example, we verify a typical lifecycle: a trip is created, goes through \texttt{Recording}, \texttt{Processing}, waits for \texttt{Validation}, and finally reaches the \texttt{Published} state.

\begin{lstlisting}[language=Alloy, caption={Predicate: Complete Trip Lifecycle}]
pred show[] {
    #Trip = 0
    // Eventually a trip is created
    eventually (#Trip = 1)
    // Eventually the trip reaches the Published state
    eventually (some t: Trip | t.currentStep = Published)
}
run show for 6

\textbf{Example 3: Removing False Positives}

pred falsePositiveRemoval[] {
    some t: Trip | eventually (
        t.currentStep = Validation and
        // The number of obstacles decreases in the next step
        #t.obstacles' < #t.obstacles and
        t.currentStep' = Published

\subsubsection{Assertions}

Finally, through assertions, we verify critical safety properties. The checks confirm that strictly invalid behaviors are impossible in our model.

\textbf{Assertion 1: No Skipped Steps}

We ensure that every published trip has strictly followed the mandatory lifecycle (Recording $\rightarrow$ Validation $\rightarrow$ Published).

\begin{lstlisting}[language=Alloy, caption={Assertion: Safety Check for Steps}]
assert noSkippedSteps {
    always (all t: Trip |
        t.currentStep = Published implies
        (once t.currentStep = Validation) and
        (once t.currentStep = Recording)
        )
}
check noSkippedSteps for 8
\end{lstlisting}

\textbf{Assertion 2: Immutable Published State}
